# demo-capgemini

A lightweight team task manager with a manager progress dashboard. Built as a filmed demo for Capgemini.

## The product

Two audiences, one tension:
- **Team members** (e.g. Léa, designer): want zero friction. They just want to see their tasks and check them off. If it takes more than 5 seconds to add a task, they go back to sticky notes.
- **Manager / COO** (e.g. Marc): wants a barometer, not a control panel. A rising line is enough. Visibility without a meeting.

Core tension: **simplicity vs visibility.** The manager's dashboard must be a free by-product of what the team does naturally — no extra effort from the team to feed the reporting.

## Stack

| Layer | Choice |
|---|---|
| Framework | Vite + React + TypeScript |
| Styling | Tailwind CSS v4 |
| Charts | Recharts |
| Tests | Vitest + @testing-library/react |
| Data | Fake JSON backend: `src/data/tasks.json` via `src/data/store.ts` |
| Backend | None — React holds runtime state |

**Data rule:** components never import `tasks.json` directly. All reads and writes go through `src/data/store.ts`.

## Method

- **Shape Up** for scoping: pitches with Problem / Appetite / Solution / Rabbit Holes / No-gos. Cupcake first.
- **TDD** with Vitest: tests written before implementation. Tests are the source of truth.
- **Trunk-based** development: small atomic commits, each deployable.
- **Human checkpoints** at each step: no autonomous merges or pushes.

## v1 scope (cupcake) — what's in

- Task list with 3 stages: `todo` / `in-progress` / `done`
- Move a task between stages
- Manager dashboard: count of in-progress tasks, count of tasks completed this week, a weekly progress line chart (Recharts)

## v1 scope — what's out (explicit No-gos)

- Sub-tasks and dependencies
- Roles and permissions
- Third-party integrations (Slack, email, calendars)
- Filters, labels, priorities, assignees
- Real backend or authentication

## Folder structure

```
src/
  data/
    tasks.json       # seed data (12 tasks, 3 stages, ~3 weeks of history)
    store.ts         # getTasks, updateTaskStage, weeklyProgress
  components/        # UI components built during the demo
discovery/
  01_client_request.md
  02_manager_brief.md
  03_market_study.md
  04_user_interviews.md
  arthur-use-cases.pptx
  pitches/           # Shape Up pitches generated by /pitch
.claude/
  skills/            # product-manager, orchestrator, architect, tester, implementer, reviewer, ui-ux-pro-max
  agents/            # skeptic, user-advocate, competitor, optimist
  commands/          # challenge, scope-hammer, pitch, add, update-index
```

## Available skills

| Skill | Role |
|---|---|
| `product-manager` | Clarifies the real need, challenges assumptions, bridges design and engineering |
| `orchestrator` | Plans the Shape Up pipeline, spawns teammates, manages checkpoints |
| `architect` | Produces a technical plan with files, deps, risks, and test strategy. Never codes |
| `tester` | Writes Vitest tests before implementation (TDD) |
| `implementer` | Codes until tests pass. Never modifies tests |
| `reviewer` | Reviews diffs before PR. Read-only |
| `ui-ux-pro-max` | UI/UX design intelligence — styles, palettes, typography, chart types |

## Available agents

| Agent | Role |
|---|---|
| `skeptic` | Finds reasons an idea will fail |
| `user-advocate` | Challenges whether users actually want this |
| `competitor` | Maps the competitive landscape |
| `optimist` | Identifies the best-case scenario and conditions for success |

## Available commands

| Command | Usage |
|---|---|
| `/challenge` | Run all 4 agents in parallel on a hypothesis or file |
| `/scope-hammer` | Cut a feature to its cupcake using Shape Up's fixed-time variable-scope |
| `/pitch` | Generate a Shape Up pitch and write it to `discovery/pitches/` |
| `/add` | Extract insights from a source and merge into `discovery/` |
| `/update-index` | Rebuild `discovery/00-INDEX.md` |

## Conventions

- **Language:** English everywhere (code, comments, commits, docs)
- **Commits:** conventional commits — `feat:`, `fix:`, `test:`, `refactor:`
- **No dead code:** no `console.log`, no commented-out blocks
- **Components:** in `src/components/`
- **Data access:** only through `src/data/store.ts`
